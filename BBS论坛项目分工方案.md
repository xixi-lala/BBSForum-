# BBS论坛系统 项目分工方案

---

## 一、项目概述

**项目名称**：BBS论坛系统
**团队人数**：5人
**总分值**：80分
**开发模式**：每人全栈开发（前端页面 + 后端接口 + 数据库建表）

### 题目要求（8项基础功能）

| 序号 | 功能 |
|------|------|
| 1 | 发布帖子、回复帖子 |
| 2 | 置顶（分板块） |
| 3 | 帖子加精 |
| 4 | 发布需求信息（设置积分奖励） |
| 5 | 管理员或文章作者对文章进行修改 |
| 6 | 注册为BBS用户，维护个人资料（联系方式、工作性质、工作地点等） |
| 7 | 管理员设置板块 |
| 8 | 分板块展示 |

## 二、角色与模块分工

### 组长：帖子核心功能 + 公共架构（22%）

| 开发层面 | 具体内容 |
|----------|----------|
| **数据库** | 帖子表（posts）、回复表（replies）建表，设计全部表结构 |
| **后端接口** | 发布帖子、回复帖子、帖子列表查询、帖子详情查询、删除帖子/回复 |
| **前端页面** | 发帖页面、帖子详情页（含回复列表）、首页帖子列表 |
| **公共架构** | 项目初始化、目录结构、导航栏、底部栏、全局CSS样式、路由配置、代码合并 |

**涉及文件清单**：

```

  reply.js           # 回复相关路由

    list.html        # 帖子列表页
```

### 5.2 核心技术栈建议

| 类别 | 推荐方案 |
|------|----------|
| 后端框架 | Express（Node.js）或 Flask（Python） |
| 模板引擎 | EJS / Pug（Node.js）或 Jinja2（Python） |

- 颜色方案（主色调、辅助色、背景色）

## 六、核心数据表设计（组长主导，全员确认）

### 用户表（users）—— 组员B负责

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键自增 |
| username | VARCHAR(50) | 用户名，唯一 |
| password | VARCHAR(255) | 加密密码 |
| phone | VARCHAR(20) | 联系方式 |
| job_type | VARCHAR(50) | 工作性质 |
| job_location | VARCHAR(100) | 工作地点 |
| role | ENUM('user','admin') | 角色，默认user |
| created_at | DATETIME | 注册时间 |

### 板块表（categories）—— 组员C负责

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键自增 |
| name | VARCHAR(50) | 板块名称 |
| description | VARCHAR(200) | 板块描述 |
| sort_order | INT | 排序权重 |
| created_at | DATETIME | 创建时间 |

### 帖子表（posts）—— 组长+组员A负责

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键自增 |
| title | VARCHAR(100) | 标题 |
| content | TEXT | 内容 |
| user_id | INT | 作者ID（外键） |
| category_id | INT | 所属板块ID（外键） |
| is_top | TINYINT | 是否置顶，默认0（组员A） |
| is_elite | TINYINT | 是否加精，默认0（组员A） |
| view_count | INT | 浏览次数，默认0 |
| created_at | DATETIME | 发布时间 |
| updated_at | DATETIME | 修改时间 |

### 回复表（replies）—— 组长负责

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键自增 |
| content | TEXT | 回复内容 |
| user_id | INT | 回复者ID（外键） |
| post_id | INT | 所属帖子ID（外键） |
| created_at | DATETIME | 回复时间 |

### 需求表（demands）—— 组员D负责

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键自增 |
| title | VARCHAR(100) | 需求标题 |
| content | TEXT | 需求描述 |
| user_id | INT | 发布者ID（外键） |
| category_id | INT | 所属板块ID（外键） |
| score | INT | 悬赏积分 |
| status | ENUM('open','closed') | 状态，默认open |
| best_reply_id | INT | 最佳回复ID（采纳后填入） |
| created_at | DATETIME | 发布时间 |

### 积分流水表（score_logs）—— 组员D负责

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键自增 |
| user_id | INT | 用户ID |
| score | INT | 积分变动（正数为获得，负数为扣除） |
| reason | VARCHAR(100) | 变动原因 |
| created_at | DATETIME | 时间 |

## 八、依赖关系与开发顺序

### 依赖图

```
组员C（板块表）
    |
    +--- 组长（帖子核心，发帖要选板块）
    |       |
    |       +--- 组员A（置顶加精，依赖帖子表 is_top/is_elite 字段）
    |
    +--- 组员D（需求信息，依赖板块+帖子模式）
    |
    +--- 组员B（用户系统，贯穿全程，注册登录是入口）
```

### 时间节点

| 时间 | 负责人 | 里程碑 | 产出 |
|------|--------|--------|------|
| 第1-2天 | 组长 | 架构搭建 | 项目模板、数据库SQL脚本、公共布局页面 |
| 第2天 | 组员C | 板块先行 | 板块表建表完成，板块CRUD接口可用 |
| 第3-6天 | 全员 | 并行开发 | 各自模块完整全栈开发 |
| 第7-8天 | 全员 | 联调测试 | 接口对接、bug修复、功能串联 |
| 第9-10天 | 组长 | 收尾完善 | 代码合并、风格统一、文档整理 |

### 关键节点

> **组员C必须在第2天完成板块表和板块查询接口**，组长和组员D才能正常开发发帖/需求功能（需要选板块）。

## 九、开发规范（全员遵守）

### 代码风格
- 统一缩进：2空格（或4空格，全组投票决定）
- 命名规范：文件名用小写下划线（post_detail.html），JS变量用驼峰（postList）
- 所有接口返回 JSON 格式，前端通过 Ajax/Fetch 调用
- 中文注释，每个函数说明用途

### Git协作
- 每人一个分支，命名格式：feature/模块名
- 每天至少提交一次，推送前先拉取最新代码
- 合并到主分支前，必须先在本地测试无误

### 文档要求
- 每人写自己模块的接口文档（1页即可）
- 列出：接口URL、请求方式、请求参数、返回示例

## 十、答辩准备建议

### 演示流程（按题目顺序）

```
注册/登录 -> 浏览板块 -> 发布帖子 -> 回复帖子
-> 管理员置顶/加精 -> 编辑文章 -> 发布需求悬赏 -> 积分查看
```

### 每人准备

- **一句话模块介绍**：我负责的是XXX模块，实现了XXX功能
- **演示1-2个亮点功能**：选最有交互感的
- **技术亮点准备**：
  - 组长可讲：MVC架构设计、数据库关系设计
  - 组员A可讲：多条件排序算法（置顶>加精>时间）
  - 组员B可讲：密码加密存储、session鉴权机制
  - 组员C可讲：RBAC权限控制（作者/管理员不同权限）
  - 组员D可讲：积分流转的事务处理
